# Публикатор

Замыкает текстовый завод (Фаза 1): берёт ОДОБРЕННЫЙ владельцем пост и ставит его в нативные
«отложенные» канала на слот из контент-плана — как руками человека (обложка сверху + полный
текст), а не превью-карточкой Bot API. Telegram отправляет по расписанию; владелец видит и
правит пост в отложенных канала до отправки.

## Роль (полная, цель)
«Отдел выпуска»: единая точка, через которую готовый контент попадает в мир.
- **Сбор готового контента** к публикации (приём финала от Криейтора/владельца).
- **График** — берёт слот из контент-плана (НЕ придумывает: КОГДА — это данные/решение, КОГО
  координировать пока некого; план — в memory/post_standard §«Ритм недели», читается через
  `core.content_plan`). Сам Публикатор только ИСПОЛНЯЕТ план.
- **Постановка в отложенные** канала (сейчас — Telegram; платформо-независимо: «куда» — отдельный
  коннектор, Threads/др. встанут рядом в Фазе 2–3).
- **Реестр для Аналитика** (позже): из какого брифа/формата родился пост — чтобы Аналитик связал
  метрики с углом. Для Telegram Аналитик и так читает канал; реестр станет нужен для др. платформ.
Координацию всех процессов позже возьмёт оркестратор (отложен — некем координировать).

## Границы и безопасность
- **Только выпуск.** Не пишет/не правит тексты (Криейтор), не судит метрики (Аналитик).
- **Детерминированный, без LLM.** Пост уходит РОВНО как одобрено, без переписывания и без
  «самостоятельных» решений. Ключ Claude не нужен (`core/publisher_bot.py`, не `agent_runtime`).
- **НЕ публикует вживую** (чек-лист PLAN §6): кладёт в нативные отложенные канала + пишет владельцу
  в ЛС «готово, на <слот>». Живой контроль — у владельца в отложенных (правит/отменяет/сдвигает).
  Постановка — только по явной команде `/schedule`.

## Как публикует (MTProto, не Bot API)
Постит аккаунтом-публикатором @Amanbabai228 (userbot) — у него права владельца, лимитов Bot API
нет. Та же сессия, что у аналитики (`TELEGRAM_SESSION`): один аккаунт — read метрики + write пост.
Коннектор: `connectors/telegram_publish/publish.py`. Выбор формы — автоматом по длине:
- короткий пост (≤ лимита подписи 1024 / 2048 с Premium) → **фото + подпись**, одним сообщением;
- длинный (флагман 2800–4096) → **обложка крупным превью НАД текстом**, одним сообщением
  (подпись бы не вместила; обложка заливается на хостинг — превью тянется по публичному URL);
- сбой/нет URL → **фолбэк**: фото отдельным сообщением + полный текст (пост не теряем).

## Контент-план (КОГДА постим)
Зеркало memory/post_standard §«Ритм недели» в `core/content_plan.py` (держать в синхроне; позже
вынесем в структурированный файл-план): **Вт + Чт** — флагман (Ф1, окно 16–19:00), **Пн/Ср/Пт** —
короткий (Ф5, окно 14–19:00), **Сб/Вс** — нет. Формат поста определяется по длине (длинный ⇒
флагман); точное время в окне и часовой пояс — в `.env`. Публикатор берёт ближайший подходящий слот,
пропуская дни, где уже стоит отложенный пост (читает отложенные канала).

## Поток
Криейтор отдаёт текст + обложку → владелец правит/одобряет → пересылает финал и обложку
Публикатору → `/preview` (что и КОГДА уйдёт) → `/schedule` → пост в отложенных канала + уведомление
владельцу → Telegram отправляет по расписанию.

## Команды
- (текст) — финальный текст поста; (фото) — обложка (подпись к фото = текст, если прислать вместе)
- `/preview` — показать, что и на какой СЛОТ уйдёт (текст — дословно)
- `/schedule [флагман|короткий]` — поставить в отложенные канала (формат по умолч. — по длине;
  гейт + защита от двойного тапа)
- `/check` — предполётная проверка: аккаунт-публикатор, Premium, часовой пояс, виден ли канал
- `/discard` — сбросить staged-пост · `/start`, `/help` — справка

## Эксплуатация
Запуск: `python run_publisher.py`. Нужны в `.env`: `PUBLISHER_BOT_TOKEN`, `PUBLISH_CHANNEL`,
`TELEGRAM_SESSION`. Опционально: `PUBLISH_PREMIUM` (1 если у аккаунта Premium),
`PUBLISH_LONG_MODE` (`preview` | `split`), `PUBLISH_UTC_OFFSET` (по умолч. +3),
`PUBLISH_FLAGSHIP_TIME`/`PUBLISH_SHORT_TIME` (по умолч. 17:00/15:00). Staged-пост —
`data/publisher_pending.json` (вне git), переживает рестарт.
