# TG-Agents — контекст проекта для Claude Code

## Что это
Персональная мульти-агентная ИИ-команда (контент-завод) вокруг Telegram-канала
и бренда владельца. Полный чертёж — в docs/PLAN.md, читай его для полной картины.

## Контекст прошлых сессий
Важные обсуждения и инсайты прошлых разговоров — в `notes/` (gitignore, локально). **Claude: читай
`notes/` при старте**, чтобы быть в теме и не переспрашивать. Архитектурные решения продублированы в docs/PLAN.md.

## Архитектура (коротко)
- Агенты-специалисты с ролями: стратег, аналитик, адаптер, продюсер, ресёрчер, разработчик.
- Память — ОТДЕЛЬНЫЙ общий слой, не внутри агентов:
  - memory/brand.md — канон (tone of voice, аудитория, цели), меняется редко
  - memory/strategy.md — живое состояние, у фактов даты, новое перекрывает старое
  - memory/decisions.md — журнал решений, только дополняется, с датами
- Принцип «мозги vs руки»: LLM-логика дёшева, интеграции (коннекторы) — самое сложное.
- Ядро отделено от коннекторов: систему переносим и на другие продукты, не только Telegram.

## Текущий фокус (с 14.06.2026)
Стартуем НЕ с выгрузки постов, а с **личного ассистента** (реальный ассистент по жизни + «живой ТуДу»), доступ через Telegram (текст; голос позже). Сначала строим общий **«движок агента» (core/)** и переиспользуем его для следующих сотрудников. Оркестратор отложен (некем координировать). Подробности и порядок — в docs/PLAN.md, §6 «Обновление 14.06.2026». Стек: Python + aiogram + Git.

## Структура репозитория (целевая)
- core/        — общий «движок агента»: бот (aiogram) + личность + доступ к памяти; переиспользуется всеми
- agents/      — определения агентов (личность/промпт, конфиг, доступы); первый — personal-assistant
- memory/      — общий слой: на старте про владельца (profile.md — канон, tasks.md — живой ТуДу, journal/ — саммари сессий и решения). Контентные brand.md/strategy.md — позже
- connectors/  — «руки»: Telegram Bot API (интерфейс), позже Metricool (аналитика/публикация), Telegram Client API/MTProto (выгрузка постов), календарь/таск-менеджеры
- tests/       — эталонные задачи для проверки агентов
- docs/        — PLAN.md и документация
- .env         — ключи (токены ботов и пр.), НИКОГДА не коммитить (держим .env.example как образец)

## Правила работы
- Объясняй понятным языком, что и зачем делаешь — владелец учится по ходу.
- Перед крупными изменениями сначала предложи план, потом выполняй.
- Секреты только в .env; никогда не пиши ключи в код или в файлы памяти.
- Идём от простого к сложному, по модулям (см. план сборки в PLAN.md).
- Каждое осмысленное изменение — отдельный git-коммит.

## Окружение
- Запуск изолированно в Docker, рабочая папка /workspace = этот репозиторий.
