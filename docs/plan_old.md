# TG-Agents — подробная спецификация проекта

> **Назначение этого документа.** Это полный, развёрнутый чертёж проекта. Его читает и владелец, и Claude Code (агент-Разработчик). Здесь зафиксированы не только задачи, но и *логика и причины* решений — чтобы не приходилось ничего дообъяснять заново. Короткая выжимка для ежедневного контекста лежит в `CLAUDE.md`; этот файл — глубокий справочник.
>
> Главный принцип всего проекта: **от самого простого к самому тяжёлому.** Не строим всё сразу, щупаем по одному модулю.

## 1. Откуда взялась идея и что мы в итоге строим

Идея родилась так: владелец увидел, как человек в Telegram собрал несколько ботов, каждый из которых подключён к своей ИИ-модели (условно один — на GPT, другой — на Claude, третий — на Gemini), задал каждому свою роль, и добавил их всех в одну группу, где они **общаются между собой и с владельцем**. Получилась команда специалистов, которая может совещаться и приходить к общему решению.

Из этого выросла цель: **персональная ИИ-команда — несколько агентов-специалистов с общей памятью**, которые умеют:
- общаться со мной (владельцем) и между собой в одном пространстве,
- работать с входящей информацией (ресёрч, анализ),
- и **делать дело параллельно**: один пишет, другой анализирует метрики, третий готовит видео — а не я один сижу руками.

**Первая задача команды — контент-завод** вокруг моего бренда и Telegram-канала. У меня уже есть регулярный контент, я вручную переделываю посты под другие площадки и хочу расширяться. Но контент-завод — это лишь **первый продукт и первый интерфейс**, а не потолок.

**Два железных принципа на будущее (не нарушать при проектировании):**
1. **Не привязываемся к Telegram.** Telegram — просто первый интерфейс. Ядро (агенты + память) должно переноситься на другие каналы и задачи.
2. **Не привязываемся к одному продукту.** Сегодня — мой канал. Завтра тем же ядром можно обслуживать другие продукты.

Поэтому архитектурное требование: **ядро (агенты + память + оркестрация) отделено от коннекторов** (Telegram, Metricool, видео-тулзы). Коннекторы — сменные модули.

## 2. Личные цели владельца (зачем мне это)

Это важно для тона и приоритетов, не только для функционала:
- Хочу не просто бота, а **навык и команду**: скидываю поток мыслей → получаю быстрый сильный ресёрч со всех сторон → даю команду → агенты расходятся и делают параллельно.
- Хочу **сам в этом разбираться** и легко осваивать новое. Поэтому: объясняй понятным языком, что и зачем делается. Я учусь по ходу.
- **Принцип «система должна делать меня сильнее, а не глупее».** Агенты могут знать больше деталей, чем я, — это нормально (как у владельца бизнеса специалисты знают свою область глубже). Но я никогда не делегирую **понимание „почему“ и финальное решение**. Стратег обязан объяснять логику, а не просто выдавать ответ; моё «одобряю» означает «я понял», а не «неважно». Где можно — система меня подтягивает, а не просто работает за меня.

Контекст по объёму, чтобы понимать масштаб: у меня Telegram-канал с приличным числом постов; подключён сервис аналитики/планирования **Metricool** (Instagram, Threads, TikTok, YouTube, Facebook); видео пока делал руками — сделал ~7, хочу выйти на **100+**.

## 3. Ключевые принципы архитектуры

### 3.1 Память — отдельный общий слой (НЕ внутри агентов)
Память лежит снаружи, все агенты к ней обращаются. Это решает сразу две главные проблемы любых ИИ-систем: «забывание/противоречия» и «перегрузку контекста».

Три типа памяти:
- **Канон** — стабильные факты: tone of voice, портрет аудитории, цели бренда. Меняется редко. Файл: `memory/brand.md`.
- **Живое состояние** — текущая стратегия, свежие метрики, актуальные приоритеты. У каждого факта **дата**, и **новое перекрывает старое**. Это не «противоречие», а обновление состояния. Файл: `memory/strategy.md`.
- **Журнал событий/решений** — «такого-то числа решили то-то». **Только дополняется**, всегда с датой. Это протокол сессий. Файл: `memory/decisions.md`.

**Как разруливаются противоречия.** Настоящий конфликт (а не просто «свежее перекрыло старое») не угадывается автоматически — его выносит на решение владельцу **Стратег-куратор**. Пример: метрика обновилась — это живое состояние, новое побеждает молча. А вот «два решения, которые исключают друг друга» — это уже к владельцу.

**Как лечится перегрузка контекста.** Через **retrieval**: агент тянет в себя не всю базу, а только релевантный под задачу кусок. База снаружи может быть сколь угодно большой — «в голове» агента всегда чисто, поэтому он не тупеет. На старте память — это простые .md-файлы (их можно открыть и проверить руками). **Векторный архив** контента добавляем позже, когда постов/материалов станет много.

### 3.2 Мозги vs Руки (важнейшее различие для оценки сложности)
- **Мозги** — рассуждать, переписывать пост, анализировать метрики, писать сценарий, решать. Это просто модель + промпт + память. **Дёшево и легко.**
- **Руки** — выгрузить посты, забрать метрики, опубликовать, сгенерить видео. Это интеграции/коннекторы. **Здесь 80% времени, денег и поломок.**
- Вывод: собираем по модулям, и каждую «руку» сначала гоняем вручную, потом автоматизируем. Не недооцениваем плумбинг.

### 3.3 Координация агентов
Реальное «совещание» агентов идёт через **общую память + оркестратор**, а не через болтовню в чате. Чат (в Telegram или ином интерфейсе) — это место, где **владелец смотрит и рулит**, а не основной механизм связи. Так надёжнее для продакшена.

(Технически: в Telegram с весны 2026 боты *умеют* переписываться между собой — Bot API 10.0 ввёл bot-to-bot режим и «гостевых ботов». Но для надёжности межагентную координацию держим на оркестраторе/памяти, а Telegram оставляем как человеческий интерфейс. Если делать болтовню ботов — обязательна защита от зацикливания: лимит реплик на одну дискуссию.)

## 4. Состав команды (подробно)

Принцип по моделям: **дорогая модель только на Стратега и Разработчика; воркеры — на дешёвых моделях.** Так система не разоряет по токенам.

### Стратег-оркестратор
- **Роль:** «холодный решала». Сводит выводы всех агентов, принимает решения, разруливает противоречия, запускает куратора памяти.
- **Модель:** топовая (Opus / 5-я линейка).
- **Мозги/Руки:** чистые Мозги.
- **Важно:** объясняет владельцу логику решений (см. §2, принцип «сильнее, а не глупее»).

### Личный ассистент
- **Роль:** мой интерфейс. Я надиктовываю/пишу идеи — он документирует их в память (`strategy.md`/`decisions.md`) и участвует в обсуждении, подключая остальных.
- **Модель:** средняя.
- **Мозги/Руки:** Мозги.

### Админ канала
- **Роль:** выгружает **ВСЕ** посты Telegram-канала, ведёт базу постов, ловит дубли, видит, что можно усилить/переиспользовать.
- **Модель:** средняя.
- **Мозги/Руки:** Мозги + **Руки** (выгрузка).
- **Технический нюанс:** полную историю канала нельзя получить через обычного чат-бота (история проскакивает). Нужен **Telegram Client API (MTProto)** — отдельная библиотека (напр. Telethon). Один раз заливаем весь архив в базу, дальше доливаем новое инкрементально.

### Аналитик
- **Роль:** читает метрики по всем площадкам, говорит, какой контент заходит, рекомендует.
- **Модель:** средняя.
- **Мозги/Руки:** Мозги + **Руки** (Metricool API).
- **Нюанс:** данные берёт из Metricool API (см. §5). Это закрывает аналитику по IG/YT/TikTok/Threads/FB одним источником.

### Адаптер / Дистрибьютор
- **Роль:** 1 пост → N форматов под площадки (например, пост из Telegram → ~4 поста под Threads), планирует и публикует.
- **Модель:** дешёвая.
- **Мозги/Руки:** Мозги (переписывание — тривиально) + **Руки** (Metricool API для планирования и публикации).
- **Нюанс:** публикуем **через Metricool**, а не через прямые интеграции с каждой площадкой — это сильно упрощает (одна «рука» вместо пяти) и не требует antidetect-браузеров.

### Продюсер видео
- **Роль:** длинный пост (например, лонгрид ~4000 символов) → сценарий → видео (ролик ~1–2 минуты, где подробно объясняется тема).
- **Модель:** дешёвая (для сценария).
- **Мозги/Руки:** Мозги (сценарий — легко) + **Руки** (API видео-тулзы).
- **Подход к видео:** НЕ дженерик-фейслесс со стоковыми кадрами (это «слоп», который не зашёл). Цель — **собственные футажи владельца + клон голоса** (заливаешь отснятый материал и образцы голоса → система собирает «тебя говорящего» под сценарий). Категория инструментов: HeyGen / Synthesia / Argil (у некоторых есть API для автоматизации). Это самый тяжёлый и экспериментальный модуль — делаем последним. Цель масштаба: **с 7 до 100+ роликов.** Узкое место было не в идеях, а в продакшене — этот модуль его снимает.

### Ресёрчер
- **Роль:** под обнаруженные пробелы ищет сервисы/API/готовые навыки, проверяет цену, демку, пригодность, предлагает варианты.
- **Модель:** средняя.
- **Мозги/Руки:** Мозги (+ веб-поиск, когда будет доступ).

### Разработчик
- **Роль:** строит и поддерживает саму систему; находит и подключает новые навыки/инструменты (MCP-серверы, Agent Skills); ведёт версии и откаты.
- **Модель:** Opus 4.8.
- **Мозги/Руки:** Мозги.
- **Примечание:** на практике эту роль сейчас исполняет **Claude Code** в изолированном Docker-контейнере (см. §9).

## 5. Инфраструктура и коннекторы (подробно)

### Память
- Старт: простые файлы `memory/brand.md`, `memory/strategy.md` (с датами), `memory/decisions.md` (журнал).
- Позже: векторный архив контента для семантического поиска (когда материалов много).

### Metricool API (главный коннектор «рук»)
- Закрывает **сразу две самые тяжёлые руки**: **публикацию** и **аналитику** по IG / YT / TikTok / Threads / FB.
- У Metricool есть **REST API** и для планирования/публикации, и для аналитики (пост-уровневые метрики, лучший контент по каждой сети, лучшее время постинга).
- ⚠️ **API доступен только на тарифе Advanced и выше** — заложить в бюджет.
- Бонус: есть встроенный **мониторинг конкурентов** (легальный срез трендов) и интеграции с **Zapier / Make** (часть связок можно собрать без кода).

### Telegram Client API (MTProto)
- Для полной выгрузки истории канала (Админ). Это не Bot API. Библиотека уровня Telethon.

### Видео-тулза API
- Для Продюсера: footage + клон голоса (HeyGen / Synthesia / Argil или аналог с API).

### Принцип
- **Ядро отделено от коннекторов** — чтобы переносить систему на другие интерфейсы и продукты.

## 6. Развитие, защита и расширение агентов

Агент = его **промпт + память + навыки**. Всё это можно улучшать — но улучшение можно и сломать. Поэтому два механизма: защита/восстановление и управляемое расширение.

### 6.1 Версионирование и восстановление (защита)
- Разделяем **определение агента** (промпт, навыки, конфиг) и **память** (накопленные знания).
  - Определение → под версионным контролем (**Git**). Это «резервная база»: любая правка — новая версия, всегда можно откатиться к последней рабочей. Сломал улучшением → откат за секунды.
  - Память → регулярные бэкапы-снимки, отдельно от определения.
- **Гейт перед внедрением (тест):** у каждого агента — набор «эталонных задач» (`tests/`). Новую версию сначала прогоняем на них. Стало хуже — не выкатываем.
- **Не мутируем живого:** улучшаем копию → сравниваем → только потом промоутим в боевого (staging / canary).

### 6.2 Расширение навыков (петля добычи)
Уже существующие стандарты, на которые опираемся:
- **MCP (Model Context Protocol)** — открытый стандарт: подключать агенту новые инструменты/доступы без переписывания. Наши коннекторы — это оно.
- **Agent Skills (SKILL.md)** — навык = папка с инструкцией (+ опционально код), учит агента конкретному типу задач. Хранится в репозитории, шарится, подгружается по необходимости.

**Петля «нашли пробел → добыли навык»:**
1. Оркестратор замечает: под задачу нет инструмента/навыка.
2. **Ресёрчер** ищет варианты (сервис / API / готовый skill), проверяет цену, демку, пригодность.
3. **Разработчик** устанавливает/подключает (MCP-сервер или skill), тестирует.
4. Прошло гейт → навык становится частью команды.

### 6.3 Безопасность
- Тянуть произвольный код/skills с GitHub прямо в агентов — **дыра в безопасности** (зловредные навыки, prompt injection, supply chain).
- Поэтому: новые навыки/код ставятся **только с подтверждения владельца**, источники проверяются, по возможности — песочница. Ресёрчер предлагает — человек одобряет.
- Полностью автономную самоустановку оставляем на «когда дозреет и будет безопасно».

## 7. План сборки: от простого к сложному

> Щупаем по одному модулю. У каждого этапа — цель и критерий готовности.

- **Этап 0 — Сегодня, без кода.** Обкатать сценарий вручную на Проектах Claude: проекты-специалисты + проект «Стратег», куда скидываю выжимки. Цель — понять, какой обмен между ролями реально полезен. _Критерий: понятно, кто кому что передаёт._
- **Этап 1 — Фундамент.** Общая память (`memory/*.md`) + **выгрузка всех постов канала** (Админ, через MTProto) в базу. _С самого начала держим определения агентов и память под версионным контролем (Git) — страховка из §6._ _Критерий: весь архив постов доступен и без дублей._
- **Этап 2 — Быстрая победа.** Адаптер/репёрпасинг как чистый LLM. Публикую пока руками. _Критерий: дедуп и tone of voice из памяти работают, выдаёт адекватные версии под площадки._
- **Этап 3 — Аналитик.** Метрики через **Metricool API**. _Критерий: видит метрики по всем площадкам и даёт рекомендации._
- **Этап 4 — Автопубликация.** Адаптер постит/планирует через Metricool API (часть можно через Zapier/Make). _Критерий: публикация без ручного копирования._
- **Этап 5 — Оркестратор + Личный ассистент.** Агенты болтают между собой/координируются; я надиктовываю — ассистент пишет в память и подключает остальных. _Критерий: одна команда от меня → параллельная работа нескольких агентов._
- **Этап 6 — Продюсер видео.** Сценарий → видео (footage + клон голоса). _Критерий: путь от поста до готового ролика без ручного монтажа; масштаб к 100+._
- **Этап 7 — Петля улучшения.** Эталонные тесты на каждого агента + петля добычи навыков (Ресёрчер находит → владелец одобряет → Разработчик ставит и тестит).

## 8. Backlog (отложенные идеи — не теряем)

- **Синтетический аватар.** Скан лица → цифровой двойник. Готовиться можно уже сейчас (собирать футажи/голос), внедрять — когда качество дозреет.
- **Threads-тренд-бот.** Бот читает ленту, анализирует тренды, рекомендует, сам постит. Тонкость: полноценное чтение чужого фида ограничено API Meta, прямой парсинг = хрупкий скрейпинг. **Чистая версия** — тренды через competitor-аналитику Metricool. Полный «бот читает весь фид» — отдельный рисковый проект.
- **TikTok Ads / Google Ads бот.** Отдельный модуль под платный трафик. Дорастём.
- **Полностью автономная самоустановка навыков.** Когда станет безопасно (§6.3).
- **Другие продукты.** Тем же ядром обслуживать не только канал.

## 9. Технические решения и окружение разработки

- **Среда разработки:** Claude Code запущен **в изолированном Docker-контейнере**, рабочая папка `/workspace` = этот репозиторий. Контейнеру смонтирована **только** папка проекта — доступа к остальному ПК владельца у агента нет (принцип least privilege). Логин — на подписке Claude Max (без поштучных API-токенов на этапе разработки).
- **Версионирование:** Git с самого начала. Каждое осмысленное изменение — отдельный коммит. Репозиторий: `TG-Agents`.
- **Структура репозитория (целевая):**
  - `agents/` — определения агентов (промпты, конфиги)
  - `memory/` — `brand.md`, `strategy.md`, `decisions.md`
  - `connectors/` — Metricool, Telegram Client API, видео
  - `tests/` — эталонные задачи (гейт перед внедрением)
  - `docs/` — этот файл (PLAN.md) и документация
  - `.env` — ключи, **НИКОГДА не коммитить**; держим `.env.example` как образец
- **Стек:** пока не зафиксирован; с учётом окружения владельца (PyCharm, Python, Docker, опыт деплоя на Hetzner) Python — вероятный кандидат. Решается с Разработчиком. _Открытый вопрос._
- **Рабочий поток (как владелец работает):** стратегию/развилки/ресёрч обсуждаем в чате Claude.ai («кабинет стратега»); реализацию делает Claude Code в репо («мастерская»). Решения принимаются в чате — выполняются в коде.

## 10. Деньги (чтобы цифры не врали)

- **Два разных кошелька.** Подписки (ChatGPT Plus, Claude Pro/Max) — это чат-приложения и Claude Code. Они **не** дают доступ к API. Автоматические боты в продакшене работают на **API-ключах** (оплата по токенам, отдельно и сверху).
- **Цены API (вход/выход за 1M токенов):** Haiku 4.5 — $1/$5, Sonnet 4.6 — $3/$15, Opus 4.8 — $5/$25.
- **Рычаги экономии:** дешёвые модели на воркеров + дорогая только на Стратега; кэширование промптов (память кэшируется) режет стоимость кэшированного входа на ~90%.
- **Грубая прикидка:** при работе несколько раз в день — ориентир ~$30–80/мес API сверху подписок. Плюс Metricool Advanced (ради API). Плюс подписка на видео-тулзу.

## 11. Глоссарий

- **MCP (Model Context Protocol)** — открытый стандарт подключения инструментов/доступов к агенту без переписывания кода.
- **Agent Skills (SKILL.md)** — переносимый «навык»: папка с инструкцией и опционально кодом.
- **MTProto / Telegram Client API** — протокол/API для полного доступа к Telegram как клиент (не бот); нужен для выгрузки истории канала.
- **RAG / retrieval** — подтягивание в контекст только релевантного куска из большой внешней базы.
- **Staging / canary** — обкатка новой версии на копии перед выкаткой в боевую.
- **Least privilege** — давать минимально необходимый доступ; основа изоляции в Docker.
